# 技术架构

## 项目概述

中汇文具 20 周年橱窗大赛系统是一个面向终端门店的全栈 Web 应用，用于举办"橱窗陈列大赛"活动。系统支持参赛者在线报名上传橱窗作品、公众投票、四位评委分板块评分、以及管理后台审核与数据导出等完整业务流程。

### 状态管理方案

项目采用轻量级状态管理，未引入 Redux / Zustand 等第三方状态库：

| 方案 | 作用域 | 说明 |
|------|--------|------|
| `useState` | 组件内 | 表单、UI 状态、分页等局部状态 |
| `useAuth` Hook | 全局 (sessionStorage) | JWT token 与角色存取，登录/登出操作 |
| `useVoteStatus` Hook | 页面级 | 轮询投票通道状态 (open/closed/not_started) |
| `sessionStorage` | 浏览器会话 | 持久化 `admin_token` 和 `admin_role` |
| axios 拦截器 | 全局请求 | 自动注入 `Authorization: Bearer <token>` 请求头 |

### 认证机制

```
客户端请求
    │
    ▼
HTTPBearer 提取 Authorization Header
    │
    ▼
python-jose 解码 JWT (HS256, 有效期 24h)
    │
    ├── 失败 → 401 Token 已过期或无效
    │
    ▼
查询 AdminUser (status != locked)
    │
    ├── 不存在/已锁定 → 401 账户不存在或已锁定
    │
    ▼
require_role("super_admin") 角色校验
    │
    ├── 不匹配 → 403 无权限访问
    │
    ▼
注入 current_user 到路由处理函数
```

**登录安全策略：**

- 密码使用 `bcrypt` 加盐哈希存储
- 连续登录失败 **5 次**后账户自动锁定 (status = locked)
- 锁定的账户无法登录 (返回 423)，需 super_admin 通过重置密码解锁
- 登录成功后 `failed_login_count` 归零

### 评分体系

总分 = A 品牌分 + B 视觉分 + C 陈列分 + D 执行分 + 人气分 = **满分 20 分**

| 板块 | 评委角色 | 满分 | 评分项 |
|------|----------|------|--------|
| **A - 品牌与活动规范** | scorer_a | 4 分 | 官方海报规范展示 (0/0.5/1)、MP 产品陈列占比 (0/0.5/1)、指定新品露出 (0/0.5/1)、品牌识别度 (0/0.5/1) |
| **B - 视觉设计表现** | scorer_b | 5 分 | 色彩搭配与整体美感 (0/1/2)、橱窗创意与主题表达 (0/1/2)、空间布局与层次感 (0/0.5/1) |
| **C - 陈列专业度** | scorer_c | 4 分 | 产品陈列逻辑 (0/1/2)、主推产品突出程度 (0/0.5/1)、价格/信息展示清晰度 (0/0.5/1) |
| **D - 执行与细节** | scorer_d | 2 分 | 橱窗整洁度与维护 (0/0.5/1)、灯光与展示效果 (0/0.5/1) |
| **人气分** | 系统自动 | 5 分 | 根据票数按阶梯计算 |

**人气分阶梯 (默认规则)：**

| 票数范围 | 人气分 |
|----------|--------|
| 0 - 100 | 1 分 |
| 101 - 300 | 2 分 |
| 301 - 500 | 3 分 |
| 501 - 999 | 4 分 |
| >= 1000 | 5 分 |

> 人气分阶梯可通过 `site_configs` 表中的 `popularity_score_config` 配置项动态调整。

**评分流程：**
1. 每位评分员登录后只能看到自己板块的评分任务列表
2. 对每个已审核通过的作品逐项打分，系统自动计算板块小计
3. 评分可多次修改，直到 super_admin 执行"锁定"操作
4. 锁定后评分不可修改 (status = locked, 返回 423)

## 部署架构

### Docker 构建流程 (Dockerfile)

```
┌─────────────────────────────────────────────┐
│              构建阶段 (Docker Build)          │
├─────────────────────────────────────────────┤
│ 1. 基础镜像: python:3.12-slim               │
│ 2. 安装 Node.js 20.x (用于前端构建)          │
│ 3. npm ci --silent (安装前端依赖)             │
│ 4. npm run build (Vite 编译 → dist/)        │
│ 5. pip install -r requirements.txt           │
│ 6. cp frontend/dist → /app/static            │
├─────────────────────────────────────────────┤
│              运行阶段                         │
├─────────────────────────────────────────────┤
│ CMD: bash startup.sh                         │
│      └→ uvicorn app.main:app                │
│         --host 0.0.0.0 --port $PORT          │
└─────────────────────────────────────────────┘
```

### 静态资源与 SPA 路由

- Vite 构建配置 `base: "/static/"`，所有前端资源路径以 `/static/` 为前缀
- FastAPI 将 `/static` 挂载为 StaticFiles 目录
- 所有未匹配 API 的路由 (`/{full_path:path}`) 返回 `index.html`，由前端 react-router-dom 接管客户端路由

### Railway 部署配置 (railway.json)

```json
{
  "build": {
    "builder": "DOCKERFILE",
    "dockerfilePath": "Dockerfile"
  },
  "deploy": {
    "startCommand": "bash startup.sh",
    "healthcheckPath": "/api/health",
    "healthcheckTimeout": 60
  }
}
```

- 使用 Dockerfile 构建方式部署
- 健康检查端点: `GET /api/health`
- 环境变量通过 Railway 平台注入 (.env 配置项)
- 单实例运行，前后端同一进程
