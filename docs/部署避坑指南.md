# 部署避坑指南

> 记录 2026-06-26 部署过程中踩过的所有坑及解决方案，确保下次部署不再走弯路。

## 坑 1：alembic 迁移在 SQLite 下崩溃

**现象：** 容器启动日志显示 `MissingGreenlet: greenlet_spawn has not been called`

**原因：** `alembic/env.py` 把 `sqlite+aiosqlite:///...` URL 原样传给同步的 `engine_from_config`，但 `aiosqlite` 驱动只能在 async 上下文里使用，同步环境下直接报 `MissingGreenlet`。

**修复：** [alembic/env.py](/Users/wangshengyu/Documents/工作/MP CRM/20周年橱窗大赛/20周年橱窗活动vibecoding/backend/alembic/env.py:14)

```python
# 之前
config.set_main_option("sqlalchemy.url", _to_async_url(settings.database_url).replace("+asyncpg", ""))

# 之后 — 对 SQLite 也去掉 async driver 前缀
config.set_main_option("sqlalchemy.url", _to_async_url(settings.database_url).replace("+asyncpg", "").replace("+aiosqlite", ""))
```

**教训：** alembic 是同步工具，无论用什么数据库，env.py 里传给它的 URL 必须是同步 driver。

## 坑 2：supabase-py v2 中 `SupabaseException` 不存在

**现象：** `ImportError: cannot import name 'SupabaseException' from 'supabase'`

**原因：** 新版 supabase-py 改了类名。`requirements.txt` 里的 `supabase` 没锁版本，Railway 装到了 v2.x，而代码 import 了 v1.x 才有的 `SupabaseException`。

**修复：** [storage.py](/Users/wangshengyu/Documents/工作/MP CRM/20周年橱窗大赛/20周年橱窗活动vibecoding/backend/app/services/storage.py:3)

```python
# 之前
from supabase import create_client, SupabaseException
...
except SupabaseException:

# 之后 — 用通用 Exception，反正都是降级逻辑
from supabase import create_client
...
except Exception:
```

**教训：** 引入外部 SDK 时，锁版本（`supabase==X.Y.Z`）或者只依赖通用异常类型。不要假设类名永远不变。

## 坑 3：Docker 缓存层导致旧代码被部署

**现象：** 代码已推送到 GitHub，构建日志显示 step 全部完成，但容器里运行的仍是旧版本代码（比如第 6 行还是 `supabase = create_client(...)` 而不是懒加载版本）。

**原因：** Railway 的 Metal builder 使用了 Docker 层缓存。当 `COPY backend/ .` 这步没检测到文件变更（或者用了快照而非最新 git commit）时，会直接用缓存层，跳过新代码。

**修复：**
1. Dockerfile 里加 `ARG CACHE_BUST=1` 打散缓存
2. 不要依赖 `railway service redeploy`，**用 `railway up --yes` 直接从本地目录推送**

```bash
cd /path/to/project
git push origin master
railway up --yes
```

**教训：** GitHub → Railway 的自动部署在连续失败后会停摆。最可靠的方式是 `railway up` 从本地推送，它会把当前目录的文件原样打包上传，不经过 Docker 缓存。

## 坑 4：`backend/static` 是符号链接，Docker COPY 无法解析

**现象：** API `/api/health` 返回 200，但所有前端页面返回 404（22 字节的 FastAPI 默认 404）。

**原因：** `backend/static` 是指向 `../frontend/dist` 的符号链接。Dockerfile 里 `COPY backend/ .` 会把符号链接原样复制进容器，但链接目标 `../frontend/dist` 在容器里不存在（Dockerfile 没复制 `frontend/` 目录）。

**修复：**
```bash
rm backend/static
cp -r frontend/dist backend/static
```

同时，`.gitignore` 第 32 行的 `static/` 会阻止 `backend/static/` 被 git 跟踪。改成只忽略根目录：

```diff
- static/
+ /static/
```

然后强制添加：
```bash
git add -f backend/static .gitignore
git commit -m "fix: replace symlink with real frontend build"
```

**教训：** 符号链接在 Docker 和 Git 里都是坑——Docker 不跨目录解析，Git 不跟踪跨目录的链接。生产部署要么用真实目录，要么在 Dockerfile 里 `COPY frontend/dist/ /app/static/` 显式指定。

## 坑 5：SPA 路由没有 fallback，`/auth` 等返回 404

**现象：** 首页 `/` 返回 200，但 `/auth`、`/register`、`/admin/login` 都返回 404。

**原因：** FastAPI 的 `StaticFiles(directory=STATIC_DIR, html=True)` 只在根路径 `/` 自动返回 `index.html`。对 `/auth` 这种 SPA 路由，Starlette 会去静态目录下找名为 `auth` 的文件，找不到就 404。

**修复：** [main.py](/Users/wangshengyu/Documents/工作/MP CRM/20周年橱窗大赛/20周年橱窗活动vibecoding/backend/app/main.py:78)

不用 `app.mount("/", StaticFiles(...))` ，改为：
1. 单独 mount `/assets` 目录
2. 加一个 `/{full_path:path}` 兜底路由，对于存在的静态文件直接返回，不存在的全部返回 `index.html`

```python
if STATIC_DIR:
    # 静态资源走 /assets
    assets_dir = os.path.join(STATIC_DIR, "assets")
    if os.path.isdir(assets_dir):
        app.mount("/assets", StaticFiles(directory=assets_dir), name="assets")

    # SPA 兜底：所有非 API 路径都返回 index.html
    @app.get("/{full_path:path}")
    async def serve_spa(full_path: str):
        target = os.path.join(STATIC_DIR, full_path)
        if full_path and os.path.isfile(target):
            return FileResponse(target)
        return FileResponse(os.path.join(STATIC_DIR, "index.html"))
```

**注意：** 这个兜底路由必须放在**所有** API Router 之后注册，否则它会吃掉 `/api/*` 请求。

**教训：** HashRouter（`/#/path`）不需要 fallback，但 BrowserRouter（`/path`）必须做。本项目用了 HashRouter，但用户直接刷新或手动访问 `/auth` 时仍需要服务端返回 `index.html`。

## 坑 6：Railway 域名需要手动创建和同步

**现象：** `railway status` 显示 `● Online`，但公网 URL 返回 Railway 网关 404：`"Application not found"`（响应头 `x-railway-fallback: true`）。

**原因：** 多次失败部署后 Railway 域名被自动清理。`railway up` 重建了容器但没自动创建域名。

**修复：**
```bash
railway domain
```

这会创建服务域名并等待同步。用 `railway domain status <domain-id>` 确认状态变成 `ACTIVE`。

**教训：** 部署成功后如果公网访问不了，先检查 `railway status` 输出里有没有 `url:` 字段。没有的话 `railway domain` 创建。

## 部署检查清单

每次部署后，按顺序验证：

```bash
# 1. 确认部署状态
railway status
# 预期: ● Online，url 字段非空

# 2. 确认 API 健康检查
curl -s https://zhonghui-competition-production.up.railway.app/api/health
# 预期: {"status":"ok"}

# 3. 确认首页
curl -s -o /dev/null -w "%{http_code}" https://zhonghui-competition-production.up.railway.app/
# 预期: 200

# 4. 确认 SPA 路由
curl -s -o /dev/null -w "%{http_code}" https://zhonghui-competition-production.up.railway.app/auth
# 预期: 200

# 5. 确认静态资源
curl -s -o /dev/null -w "%{http_code}" https://zhonghui-competition-production.up.railway.app/assets/index-CU38TeXu.js
# 预期: 200（文件名可能随构建变化）

# 6. 确认管理后台
curl -s -o /dev/null -w "%{http_code}" https://zhonghui-competition-production.up.railway.app/admin/login
# 预期: 200
```

## 快速部署命令

```bash
cd /path/to/project

# 如果有前端修改
cd frontend && npm run build && cd ..
rm -rf backend/static
cp -r frontend/dist backend/static

# 提交 & 推送
git add -A && git commit -m "deploy: ..." && git push origin master

# 部署
railway up --yes

# 等待构建后如果没域名
railway domain
```
